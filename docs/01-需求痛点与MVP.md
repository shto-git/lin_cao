# 需求痛点与 MVP

## 真实痛点

从需求文档看，业务痛点集中在五处。

第一，资料太散。政策、标准、历史规划、资源数据、项目库、空间图件、统计年鉴分散在不同格式里，人工整理费时，还容易漏来源。

第二，长文本难控。5 万到 10 万字规划不能一次生成，必须按大纲、章、节、段逐层生成。否则结构会散，前后口径会冲突。

第三，专业约束强。林草规划里很多表达不能随意写，政策依据、技术规范、指标单位、年份口径、行政区划和工程名称都要统一。

第四，图表和正文容易脱节。表格、指标、投资估算、项目库和正文描述如果不是同源生成，后期改一处会牵连多处。

第五，审查闭环成本高。专家意见往往指向某一章、某类指标或某条政策，如果系统只会整篇重写，反而增加返工。

## MVP 不做什么

第一版不追求完整替代 Word，也不直接做正式报批成果。

暂缓内容：

- 在线多人协同编辑；
- 完整类 Word 排版器；
- 法定空间图件自动制图；
- 完整 GIS 分析；
- 多租户组织权限体系；
- 移动端审阅；
- 全量历史案例库治理。

这些都重要，但不是验证首轮价值的最小闭环。

## MVP 要做什么

MVP 目标是跑通“一个真实规划初稿的生产链路”：

```text
项目信息
  -> 规划类型模板
  -> 大纲和字数分配
  -> 每章检索计划
  -> 从知识库取依据
  -> 章节草稿任务
  -> 基础质检报告
  -> Markdown/Word 导出
```

最低可验收能力：

1. 创建一个规划项目，录入规划名称、区域、期限、类型和目标字数。
2. 按规划类型生成标准大纲，并把目标字数分配到章节。
3. 为每一节生成检索问题，明确要查政策、规范、基础数据还是历史案例。
4. 对接 RAGFlow 的数据集和检索接口，拿到可追溯 chunk。
5. 生成章节写作任务，任务里带上章节目标、写作约束、候选依据和风险提示。
6. 对草稿做基础质检：禁用表述、无来源事实、核心指标前后不一致、缺章缺节。
7. 输出 Markdown 初稿和质检报告，先满足内部评审，不追求最终排版。

## 首个试点规划类型

建议优先用“县域林业发展规划”做试点。

理由：

- 章节结构相对稳定；
- 数据来源通常包括资源调查、统计年鉴、上位规划和项目库，能覆盖大部分资料类型；
- 对图件有要求，但不像自然保护地总体规划那样高度依赖法定边界；
- 业务人员容易判断文本是否像真实规划。

## 关键验收口径

不要只看“生成得像不像文章”，要看下面这些：

- 每章是否知道自己该引用什么；
- 文本里出现的数据是否能找到来源；
- 同一指标在全文是否一致；
- 专业术语是否稳定；
- 缺资料时系统是否提示，而不是编一个看似合理的结论；
- 业务人员是否愿意拿这份初稿继续改，而不是推倒重写。
