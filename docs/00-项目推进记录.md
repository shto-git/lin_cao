## 当前判断

需求里最难的不是“让模型写一段规划文本”，而是把 5 万到 10 万字成果拆成可控生产流程：

- 每章要知道该查什么资料；
- 每节要知道必须引用哪些政策、数据和案例；
- 生成内容要能回溯到来源；
- 前后章节的数据、年份、指标口径不能打架；
- 评审意见要能定位到章节和段落，而不是整篇重写。

所以第一阶段不应先做一个聊天界面，也不应直接改 RAGFlow 前端。先做业务编排层更稳。

## 已定方向

采用“两层架构”：

```text
林草规划业务层
  - 项目、模板、大纲、章节任务、质检、导出
  - 管理规划编制过程和专业规则

RAGFlow 知识底座
  - 文件解析、chunk、向量/关键词检索、引用来源、数据集管理
  - 通过 REST API 调用，不直接改数据库
```

这样做的好处：

- 可以复用 RAGFlow 已经成熟的复杂文档解析和检索能力；
- 林草规划规则不会散落在 RAGFlow 内部；
- 后续如果更换知识底座，业务层仍可保留；
- 试点阶段能更快验证真实痛点。
