## 决策 1：知识底座

### 推荐方案：RAGFlow（继续沿用）

| 维度 | RAGFlow | Dify | FastGPT | LlamaIndex |
|------|---------|------|---------|------------|
| 文档解析 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| 中文支持 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐ |
| 私有化部署 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| API 易用性 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 引用追溯 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 维护成本 | 中（需 ES） | 低 | 低 | 高 |

**理由**：
1. RAGFlow 的文档解析能力（PDF/Word/Excel 多格式）是业界最好的，这对林草规划资料入库至关重要
2. 引用追溯能力天然支持，能标注每个 chunk 的来源文件和页码
3. 已有初版实现，切换成本高
4. 社区活跃（GitHub 持续更新）

**风险**：
- 依赖 Elasticsearch，部署较重
- 需要确认是否可以用简化版部署（关闭 ES 用内置向量库）

**备选方案**：如果 RAGFlow 部署太重，可降级为 **LlamaIndex + ChromaDB**（纯 Python，轻量），但文档解析需自己写

## 决策 2：LLM 调用方案

### 推荐方案：多模型路由 + 免费优先

```
任务类型          →  模型              →  理由
────────────────────────────────────────────────────
大纲生成          →  DeepSeek Chat      →  中文好，免费
检索问题生成      →  DeepSeek Chat      →  结构化输出，免费
章节草稿生成      →  GPT-5.5 (Codex)   →  代码/长文能力强
质检分析          →  DeepSeek Chat      →  规则匹配，免费
```

**理由**：
1. 章节生成是最核心、最消耗 token 的任务，值得用最强模型
2. 大纲/检索/质检都是结构化任务，DeepSeek 免费版足够
3. 通过 OpenAI-compatible API 统一接口，切换成本低

**风险**：
- GPT-5.5 通过 Codex CLI 调用，需要 OpenAI 账户（用户已有免费额度）
- 未来可能需要接入国产模型（通义千问、GLM）作为备选

**备选方案**：全部用 DeepSeek（最省钱），但章节生成质量可能略低

## 决策 3：后端框架

### 推荐方案：FastAPI（继续沿用）

| 维度 | FastAPI | Flask | Django |
|------|---------|-------|--------|
| 异步支持 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 开发效率 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| WebSocket | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| API 文档自动生成 | ✅ | ❌ | ❌ |
| 学习曲线 | 中 | 低 | 高 |

**理由**：
1. FastAPI 已有初版实现，不需要重写
2. 异步支持对 LLM 调用（长等待）非常重要
3. 自动生成 API 文档（Swagger UI）方便前端联调
4. WebSocket 支持实时进度推送（章节生成进度）

**风险**：无，已验证可用

## 决策 4：前端方案

### 推荐方案：静态 HTML + Vue 3（渐进引入）

**MVP 阶段**：继续用静态 HTML/CSS/JS（已有初版）
**Phase 2**：引入 Vue 3（通过 CDN，不用构建工具）

| 维度 | 静态 HTML | Vue 3 | React |
|------|-----------|-------|-------|
| 开发速度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 组件复用 | ⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 学习曲线 | 低 | 中 | 高 |
| 构建工具 | 无 | 可选 | 必须 |

**富文本编辑器**：
- **推荐**：TinyMCE 5（轻量、免费、功能全）
- **备选**：Quill（更轻量）
- **不推荐**：ProseMirror（太重）、Draft.js（已停更）

**图表库**：
- **推荐**：ECharts 5（百度开源，中文文档好，支持各种图表）
- **备选**：Chart.js（更轻量，但图表类型少）

**理由**：
1. 当前前端已可用，不需要重写
2. Vue 3 通过 CDN 引入，渐进增强，不需要 Node/npm
3. 林草规划系统不是交互密集型的，静态 HTML + 少量 JS 够用

**风险**：
- 静态 HTML 在功能复杂后可能难以维护 → Phase 2 引入 Vue 3 解决

## 决策 5：数据库

### 推荐方案：SQLite（MVP）→ PostgreSQL（生产）

| 维度 | SQLite | PostgreSQL |
|------|--------|------------|
| 部署复杂度 | 零（文件数据库） | 需要安装服务 |
| 并发支持 | 差 | 好 |
| 数据完整性 | 一般 | 强 |
| Python 支持 | 内置 | 需 psycopg2 |

**理由**：
1. MVP 阶段用户量小（单人/几人），SQLite 完全够用
2. 零部署复杂度，打开就能用
3. 数据迁移到 PostgreSQL 的代码改动最小（SQLAlchemy 抽象层）

**Redis**：MVP 不需要。章节生成是同步调用（用户等待看结果），不需要任务队列。

**风险**：
- SQLite 不支持并发写入 → MVP 单人使用没问题
- 未来迁移到 PostgreSQL 需要数据迁移脚本

## 决策 6：任务队列

### 推荐方案：MVP 不做，Phase 2 用 Celery + Redis

| 维度 | 无队列（同步） | Celery + RQ | 简单线程池 |
|------|--------------|-------------|-----------|
| 复杂度 | 低 | 高 | 中 |
| 实时进度 | ❌ | ✅ | 部分 |
| 失败重试 | ❌ | ✅ | ❌ |

**理由**：
1. MVP 阶段章节生成是用户主动触发的，同步等待结果即可
2. 质检是快速的规则检查，不需要异步
3. 未来如果需要批量生成多章，再引入 Celery

**风险**：
- 同步调用 LLM 时，前端需要等 30 秒 → 用 WebSocket 推送进度

## 决策 7：部署方案

### 推荐方案：Docker Compose 一键部署

```
docker-compose.yml
  ├── lin_cao_backend  (FastAPI + SQLite)
  ├── ragflow         (RAGFlow 知识底座)
  └── nginx           (前端静态文件 + 反向代理)
```

**RAGFlow 简化部署**：
- RAGFlow 完整版依赖 Elasticsearch + MySQL + MinIO
- MVP 阶段可以用 **RAGFlow 的轻量模式**（如果支持）
- 或者替换为 **ChromaDB**（纯 Python 向量库，API 更简单）

**一键启动脚本**：
```bash
# start.bat (Windows)
docker-compose up -d
echo 服务已启动: http://localhost:8000
```

**风险**：
- Docker 在 Windows 上的性能问题
- RAGFlow 镜像较大（> 5GB）

## 技术债务清单

| 债务 | 影响 | 偿还时机 |
|------|------|---------|
| 前端无框架 | 功能复杂后维护困难 | Phase 2 引入 Vue 3 |

## 选型结论

**MVP 技术栈确认**：
- ✅ FastAPI + SQLite（后端）
- ✅ RAGFlow（知识底座）
- ✅ DeepSeek Chat + GPT-5.5（LLM）
- ✅ 静态 HTML/CSS/JS（前端）
- ✅ python-docx（Phase 2 Word 导出）
- ✅ Docker Compose（部署）

**不需要改的技术决策**：全部沿用当前方案，当前技术栈选型是正确的。
需要重做的只是**需求范围**（见 PRD v2）。
