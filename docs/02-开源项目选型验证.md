# 开源项目选型验证

本次按需求相似度核对了三个项目：RAGFlow、Dify、FastGPT。

核对时间：2026-06-24

## 需求对照

需求需要的底座能力包括：

- Word、PDF、Excel 等资料解析；
- 知识库和检索增强生成；
- 引用来源可追溯；
- 支持企业或私有化部署；
- 能被业务系统通过 API 集成；
- 对复杂文档的解析质量要足够好。

需求还需要的业务能力包括：

- 林草规划项目管理；
- 规划类型模板；
- 章节级长文本编排；
- 专业术语和指标一致性校验；
- 图表、项目库、投资估算和成果导出；
- 审查意见闭环。

开源底座主要解决第一组问题，第二组问题需要自研。

## 项目对比

| 项目 | 核对结果 | 对口点 | 不足 |
| --- | --- | --- | --- |
| RAGFlow | 最接近资料知识化底座 | README 明确定位为开源 RAG 引擎；支持复杂文档理解、模板化 chunk、引用追溯、Word/Excel/PDF 等多格式资料；有 REST API | 不提供林草规划模板、章节长文编排、专业质检和成果排版闭环 |
| Dify | 通用 AI 应用平台 | 有工作流、RAG Pipeline、Agent、模型管理和观测能力；适合快速搭应用 | 文档解析不是它的最强项；更偏应用编排，不是规划成果生产系统 |
| FastGPT | 中文知识库问答和工作流更顺手 | 支持知识库、多库混用、混合检索、流程编排，中文生态友好 | 更偏问答/Agent 平台；复杂文档解析和引用治理不如 RAGFlow 对口 |

## 事实来源

- RAGFlow GitHub：https://github.com/infiniflow/ragflow
- RAGFlow README 提到复杂文档理解、模板化 chunk、引用追溯、多格式数据源和 RAG 工作流。
- Dify GitHub：https://github.com/langgenius/dify
- Dify README 定位为开源 LLM 应用开发平台，包含工作流、RAG Pipeline、Agent 和 LLMOps。
- FastGPT GitHub：https://github.com/labring/FastGPT
- FastGPT README 定位为 AI Agent 构建平台，提供知识库、混合检索、工作流和 OpenAPI。

## 选型结论

首轮不建议 fork RAGFlow 做大改。

建议方式：

```text
RAGFlow = 知识底座
林草规划业务服务 = 编制流程和专业规则
```

业务服务通过 REST API 调用 RAGFlow：

- 创建/选择数据集；
- 上传和解析资料；
- 读取 chunk 和引用来源；
- 生成章节检索上下文；
- 把来源信息写回规划章节任务和质检报告。

这样可以先验证业务价值，不把项目变成一次大型平台改造。
