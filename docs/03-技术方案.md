## 关键流程

```text
1. 项目信息录入
   输入规划名称、区域、期限、规划类型、目标字数。

2. 生成规划大纲
   按规划类型生成章节和二级标题，并分配字数。

3. 生成检索计划
   每个章节形成 3 到 5 条检索问题，标明需要政策、规范、基础数据或历史案例。

4. 调用 RAGFlow 检索
   根据项目数据集、规划类型和章节主题检索 chunk，保留 document_id、document_name、positions、similarity。

5. 生成章节任务
   每章拿到写作目标、字数、必须引用资料、风险提示，再调用大模型生成。

6. 质检
   检查无来源事实、禁用表述、指标口径冲突、章节缺失、图表引用缺失。

7. 导出
   MVP 先输出 Markdown 和质检报告，Word/PDF 作为第二阶段。
```

## RAGFlow 接口边界

业务层只依赖 REST API，不直接访问 RAGFlow 数据库。

首批需要的接口：

- `POST /api/v1/datasets`：创建知识库。
- `POST /api/v1/datasets/{dataset_id}/documents`：上传资料。
- `POST /api/v1/datasets/{dataset_id}/chunks`：触发解析。
- `POST /api/v1/retrieval`：按问题检索 chunk。
- `POST /api/v1/chats` 或 OpenAI-compatible chat：后续用于基于知识库生成。

## 长文本生成策略

不要一次生成全文。

推荐策略：

```text
全文目标
  -> 章节目标
  -> 二级标题写作任务
  -> 证据包
  -> 草稿
  -> 章节级质检
  -> 全文一致性检查
```

章节任务必须包含：

- 章节标题路径；
- 目标字数；
- 写作要求；
- 必须引用的资料类型；
- 候选证据；
- 禁止编造的数据；
- 缺资料风险。

## 质检规则优先级

第一优先级：事实和数据。

- 出现面积、比例、投资、年份、资源量时必须有来源；
- 同一指标在全文只能有一个口径，除非明确区分年份或范围；
- 缺少资料时提示补充，不生成确定结论。

第二优先级：政策和术语。

- 政策名称要完整；
- 术语使用固定表；
- 禁止模糊部门、模糊依据、模糊年份。

第三优先级：格式和编号。

- 章节编号连续；
- 图表编号连续；
- 图表引用与正文匹配。

## 首版技术栈建议

MVP：

- Python 业务服务；
- RAGFlow 做知识底座；
- Markdown 作为中间成果格式；
- `python-docx` 或 Pandoc 做后续 Word 导出；
- 前端可先用轻量管理台，确认流程后再做类 Word 编辑。

生产化阶段：

- FastAPI；
- PostgreSQL；
- Redis + 任务队列；
- 对象存储；
- RAGFlow 独立部署；
- 审计日志和项目级权限；
- 私有化部署脚本。

## 风险

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 资料不足 | 章节生成看似完整但没有依据 | 章节任务强制带来源，缺来源降级为风险提示 |
| 数据口径冲突 | 同一指标在不同章节数值不一致 | 全文指标索引和一致性检查 |
| 解析质量不稳 | PDF 表格、扫描件、图件说明解析失败 | RAGFlow 解析结果要人工可见，关键资料支持手工修正 |
| 业务层过早复杂 | 一开始就做完整协同编辑 | 先用 Markdown 和任务状态验证流程 |
| 误把问答当编制 | 系统只会回答问题，不会产出规划 | 以章节任务和成果版本为核心对象 |
